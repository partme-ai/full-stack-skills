## 2. 架构概述 (Architecture Overview) — V1

### 2.1 系统定位

**系统定位**：{Name} 管理后台 = **Claw 引擎多实例管理平台**（多实例、配置、渠道、模型、工作台）。管理后台作为外置 Web 系统，通过 **Gateway WebSocket 协议**连接多台底层 OpenClaw / ZeroClaw 实例进行管理，并承载配置基线/模板/一致性/下发、渠道与模型配置、会话与用量聚合、日志与状态观测等能力；OpenClaw / ZeroClaw 为底层引擎，技术方案为实现与接口参考。平台不承载具体业务场景能力。

**系统目标（V1）**：与 PRD §2.1～2.3 一致。

- 支持多 OpenClaw 实例的注册、分组、健康监控与批量操作。
- 配置管理可落地：基线 + 模板 + 实例三层模型，一致性检测与批量下发，可审计；**基线版本冲突处理**（乐观锁，他人已更新则提示刷新并拦截保存）、**离线实例下发策略**（立即失败 / 自动重试，等待重试可查看与取消）。
- 单实例管理覆盖 OpenClaw 全部能力（概览/对话测试/渠道/会话/Agent/技能/节点/用量/定时/配置/调试/上下文诊断/日志），并以可视化表单替代 JSON 编辑。
- 工作台：在线实例数、总会话数、总消息数、告警数及异常实例告警待办（跳转设备中心）；会话与用量跨实例聚合；场景化运营由场景包提供。
- **审计日志**：下发记录必含**操作人**；配置变更与敏感操作可追溯（PRD §5.4）。
- **企微凭证加密存储**（PRD §5.2）；**错误响应与分页契约、时区**（PRD §5.7、§5.6）为架构约束。
- 管理后台可独立于 OpenClaw 部署，支持水平扩展与多环境（开发/预发/生产）。

### 2.2 架构原则

- **领域驱动 (DDD)**：设备/实例、配置管理、渠道接入、模型服务、会话与用量观测等按限界上下文划分；核心逻辑收敛在领域层。
- **依赖倒置**：应用层与领域层不依赖基础设施；通过 **Gateway WebSocket** 访问 OpenClaw 实例，数据库与缓存在 Infrastructure。
- **配置可落地**：基线/模板存 {Name} 数据库；下发通过 **Gateway WebSocket**（config.set/config.apply）写入并触发热重载；一致性由「基线 vs 实例当前配置」计算；**不使用 management/ics 插件的 REST API**（PRD §5.7）。
- **安全与审计**：管理后台鉴权（V1 Token）；配置与密钥不明文落日志；下发记录含操作人；企微凭证加密存储、前端全掩码。

### 2.3 技术选型

| 层次 | 选型 | 说明 |
| :--- | :--- | :--- |
| **管理后台前端** | Web：Element Plus / Avue / Echarts | B 端以 Web 为主，组件与 DNA 规范一致 |
| **管理后台后端** | Spring Boot (推荐 DDD/CQRS) 或 Node.js/TypeScript | 与 OpenClaw 同栈可加快 MVP；长期复杂业务建议 Java |
| **数据库** | PostgreSQL | {Name} 管理数据：实例、分组、基线、模板、下发任务、事件消息配置等 |
| **缓存** | Redis | 会话、热点配置、聚合统计缓存（可选） |
| **OpenClaw 插件** | TypeScript | openclaw_management、openclaw_ics、openclaw_wecom_kf（企微渠道） |
| **LLM / 向量** | OpenClaw 内置 + 多提供商 | Anthropic/OpenAI/国内模型；BM25 + 向量混合检索在 OpenClaw 侧 |

### 3.2 逻辑架构 — COLA 分层

| 层次 | 职责 | {Name} 对应 |
| :--- | :--- | :--- |
| **Adapter（适配层）** | 处理外部输入输出 | Web Controller（工作台/设备中心/配置管理/模型服务/设置）；调用 OpenClaw 的客户端封装在 Infrastructure |
| **Application（应用层）** | 业务流程编排、用例 | 实例注册与分组、基线编辑与版本、一致性计算与展示、下发任务执行、渠道配置下发、会话/用量聚合查询 |
| **Domain（领域层）** | 核心业务逻辑与模型 | Instance、Baseline、ConfigTemplate、DeploymentTask、ChannelConfig；ConsistencyService、DeploymentExecutor |
| **Infrastructure（基础设施层）** | 技术实现 | PostgreSQL 仓储实现、Redis、**OpenClaw Gateway WebSocket 客户端**（按实例 WS 地址 + Token 连接，config.get/set、sessions.list、health 等）；日志采集、用量聚合、渠道回调校验等平台基础设施 |

**依赖方向**：Adapter → Application → Domain ← Infrastructure；领域层不依赖具体数据库与 OpenClaw 客户端，通过接口注入。

### 3.3 与 OpenClaw 的交互边界（与 PRD §5.7 一致）

- **管理后台不直接读写**：不访问 OpenClaw 所在机器的文件系统（openclaw.json、AGENTS.md、工作区等）。
- **以 Gateway WebSocket 协议为主**：管理后台与各 OpenClaw 实例的交互通过 **Gateway WebSocket 协议**（与官方 Control UI 一致），**不使用 management/ics 插件的 REST API**。配置读写、会话列表、健康检查等均通过 WS 方法（如 `config.get`/`config.set`/`config.apply`、`sessions.list`、`sessions.patch`、`health`、`status`）；bindings、eventMessages 通过 config 读写。
- **平台能力扩展**：渠道连通性校验、配置审计、日志聚合、实例诊断等由管理后台自有 API 承接；仅渠道回调等 HTTP 由对应插件暴露。
- **热重载**：配置类通过 config.set 写入后，由 OpenClaw 触发热重载（如 `gateway.reload.mode="hybrid"`），无需管理后台触发进程重启。

## 4. 核心 API 契约（以 Gateway WebSocket 为核心）— V1

管理后台与各 OpenClaw 实例的交互以 **Gateway WebSocket 协议**为主（与 PRD §5.7 一致）；平台侧关注配置、状态、日志、用量与运维任务，不承载具体业务场景能力。

### 5.1 Gateway WebSocket 方法

| 分类 | 方法 | 用途 | 说明 |
| :--- | :--- | :--- | :--- |
| 配置 | config.get | 读取实例当前配置 | 用于一致性计算、单实例编辑 |
| 配置 | config.set / config.apply | 写入配置并触发热重载 | 下发任务、从基线恢复、单实例保存 |
| 配置 | bindings、eventMessages | 路由与事件消息 | 通过 config 读写（结构在 openclaw.json 内） |
| 会话 | sessions.list | 会话列表 | 单实例会话查看、运行态排障 |
| 会话 | sessions.patch | 会话状态变更 | 接管/释放/转给坐席等 |
| 健康/状态 | health、status | 健康检查、版本、插件列表 | 工作台与设备中心状态展示 |

**连接与认证**：connect 时 `params.auth.token` 携带实例 Gateway Token；实例注册时在 {Name} 数据库中存储「WS 地址 + Token」（加密或脱敏），Infrastructure 层按实例建立 WS 连接并注入 Token。

### 5.2 管理后台自有 API 与扩展

| 能力 | 实现方式 | 说明 |
| :--- | :--- | :--- |
| 渠道配置校验 | 管理后台 API | 渠道凭证校验、回调连通测试、保存前预检查 |
| 日志与诊断 | 管理后台 API | 聚合日志、异常摘要、实例诊断结果 |
| 用量统计 | 管理后台定时采集 | 消息量、Token 用量、失败率等平台指标 |

### 5.3 接口契约（与 PRD §5.7、§5.6 一致）

- **错误响应**：`{ code, message, detail? }`；前端按 code 展示 PRD §2.4 规范（表单校验、操作成功/失败、网络异常、空状态）。
- **分页**：请求 `page`、`pageSize`；响应 `list`、`total`（或等价）；格式与 PRD 接口契约一致。
- **时间格式**：接口侧使用 **ISO 8601**（UTC）；前端按用户时区转换展示（PRD §5.6）。

## 5. 数据架构 (Data Architecture) — V1

### 5.1 管理后台自有库（PostgreSQL）

| 领域 | 表/用途 | 说明 |
| :--- | :--- | :--- |
| 实例管理 | instance, instance_group, instance_tag | 实例注册信息、分组、标签 |
| 配置管理 | baseline_version, config_template, deployment_task, deployment_task_result | 基线版本（含乐观锁版本号）、模板、下发任务及每实例结果；**deployment_task 必含操作人**（PRD §5.4） |
| 企微应用 | wecom_app_credential（或等价） | 企微应用凭证；存储时**加密**（如 AES-256-GCM），密钥与库分离（PRD §5.2） |
| 渠道配置 | channel_config_snapshot | 渠道凭证、回调地址、测试结果与下发快照 |
| 运行诊断 | runtime_diagnosis_snapshot | 实例诊断结果、错误摘要、修复建议 |
| 用量统计 | usage_metrics_snapshot | 消息量、Token 用量、失败率等平台指标快照 |
| 审计 | audit_log、deployment_task.operator_id | 配置变更与敏感操作日志；**下发记录必含操作人**；保存期限与界面可见性与 PRD §5.4 一致 |

基线/模板的「配置内容」建议 JSONB 或大字段存储，结构与 openclaw.json 对齐，便于 diff 与下发。

### 5.2 缓存（Redis，可选）

- 实例健康状态短 TTL 缓存，减少对多实例的频繁健康检查。
- 聚合统计（工作台概览）短 TTL 缓存。
- 会话列表/消息记录以实时查询 OpenClaw 为主，缓存按需引入。

### 5.3 数据一致性

- **基线/模板/下发记录**：强一致性，事务落库。
- **实例当前配置**：通过 management API 实时读取，与基线 diff 得到一致性状态；下发后再次读取验证或由下发结果接口返回。
- **跨实例会话/用量**：最终一致性；各实例数据源独立，管理后台聚合展示。

## 6. 安全架构 (Security Architecture) — V1

### 6.1 认证与授权

- **V1**：管理后台登录后使用 Token 或 Session；调用 OpenClaw 实例时使用各实例的 Gateway Token（存储与传递需加密或脱敏）。

### 6.2 数据安全（与 PRD §5.2 一致）

- 配置与密钥不明文写入日志；生产环境 HTTPS。
- **企微凭证加密存储**：corpSecret、token、encodingAESKey 必须加密落库（如 AES-256-GCM），密钥与库分离；**前端全掩码**（仅末四位可见）。
- 实例 Gateway Token 存库加密或脱敏；按实例调用时由 Infrastructure 注入。

### 6.3 访问控制

- 管理后台仅允许内网或 VPN 访问（视部署方式）；OpenClaw 实例的 endpoint 可为内网地址，通过管理后台所在网络可达即可。

## 7. 部署架构 (Deployment Architecture)

### 7.1 部署方式

- **{Name} 管理后台**：独立部署（单机或集群）；可容器化（Docker/K8s）；与 OpenClaw 实例无同机绑定。
- **OpenClaw Gateway**：每台实例独立部署（见技术方案 §2.3、§8.3）；支持多 Profile、多端口；生产建议 systemd/launchd 守护。
- **网络**：管理后台能访问各实例的 **Gateway WebSocket** 地址（与官方 Control UI 一致）；企微回调需公网可达，推荐 Nginx 反代 + HTTPS。

### 7.2 高可用与扩展

- 管理后台：无状态，可水平扩展；数据库与 Redis 做主从或集群。
- OpenClaw 实例：按业务需要多实例部署；会话与配置由各实例本地或共享存储承担（见 OpenClaw 文档）；管理后台仅做「配置下发」与「读状态/会话/用量」。

## 9. 关键技术方案摘要 — V1

### 9.1 配置下发流程（可落地，与 PRD §5.2、§6 一致）

1. **基线编辑与保存**：用户在「配置管理」编辑基线；保存前**乐观锁校验**（后端基线版本 > 前端持有版本则提示「配置已被他人更新，请刷新后重新编辑」并拦截保存）；保存成功后版本递增。
2. **下发任务**：选择来源（基线/模板）与目标实例列表；Application 层生成下发任务记录，**操作人必填**并写入 deployment_task。
3. **离线实例策略**：若部分实例离线，支持 **模式 A 立即失败**（整任务失败或跳过离线实例由产品约定）或 **模式 B 自动重试**（最多 N 次，如 3 次）；等待重试的任务可查看与取消。
4. **执行下发**：对每个目标实例，Infrastructure 层通过 **Gateway WebSocket** 调用 `config.set`/`config.apply` 写入配置，触发热重载；汇总每实例成功/失败写回 DeploymentTask；前端展示结果；失败可重试并记录原因。
5. **一致性计算**：定时或按需对每个实例通过 WS `config.get` 拉取配置，与当前基线版本做 diff，更新「一致性状态」与差异项数。
